# Backend Run Artifacts

## 목적

서브에이전트 오케스트레이션 결과를 최소한의 파일로 남긴다. 파일은 추적과 재개를 위한 증거이며, 문서 자체가 작업을 복잡하게 만들면 안 된다.

## 위치

```text
.agents/runs/{run_id}/
├── inputs/
│   ├── requirement-context.md
│   └── M1/
│       ├── 001-tdd-input.md
│       ├── 002-implementation-input.md
│       └── 003-review-input.md
├── outputs/
│   └── M1/
│       ├── 001-tdd-output.md
│       ├── 002-implementation-output.md
│       └── 003-review-output.md
└── checkpoints/
    └── M1/
        ├── tdd.md
        ├── implementation.md
        └── review.md
```

기존 `*.v1.md` 파일명을 쓰는 run도 허용한다. 중요한 것은 입력, 출력, 체크포인트가 서로 연결되는 것이다.

## Requirement Context

`requirement-context.md`는 작업 절차 1단계에서 먼저 만든다. 아래 템플릿의 각 섹션은 사용자가 명시했거나 사용자와 질의응답으로 확정한 내용만 쓴다. 조금이라도 모호한 항목이 있으면 2단계 Source of Truth 패키지 고정으로 넘어가지 않고 사용자에게 질문한다.

형식 보존 기준:

- 템플릿의 섹션명, 순서, 언어, 표 구조를 바꾸지 않는다.
- 섹션명을 영문으로 번역하거나 축약하지 않는다.
- 섹션을 삭제하거나 다른 섹션에 병합하지 않는다.
- 해당 없음인 섹션도 삭제하지 않고 `해당 없음`으로 확정 내용을 남긴다.

빈 값 처리 기준:

- 실제로 해당 없음이 확정된 항목은 `해당 없음`으로 쓴다.
- 아직 모르는 항목은 `미정`으로 남기지 않는다. 사용자에게 질문해 확정한다.
- 사용자가 의도적으로 나중에 결정한다고 승인한 항목만 `사용자 승인 하에 후속 결정`으로 표시한다.

작성 완료 기준:

- 템플릿의 예시 문구를 실제 확정 내용으로 바꾼다.
- 모든 표의 빈 셀을 확정 내용, `해당 없음`, 또는 `사용자 승인 하에 후속 결정` 중 하나로 채운다.
- 상태는 최초 작성 시 `Draft`로 두고, 사용자 확인을 받아 다음 단계로 넘어갈 수 있으면 `Review` 또는 `Approved`로 바꾼다.
- 2단계 Source of Truth 패키지 고정은 `requirement-context.md`가 위 기준을 만족한 뒤에만 수행한다.

## Role Input

필수 내용:

- 목표
- 명시적 제외사항
- Source of Truth 기준 경로와 고정 사유
- Source of Truth 패키지
  - Backend common Source of Truth: `docs/rules/backend/README.md`를 포함한 `docs/rules/backend/**` 전체와 고정 사유 "backend 기준 전체 준수"
  - Task Source of Truth: 사용자 요청, `requirement-context.md`, TDD, 명시 결정, 수정 루프의 reviewer output
  - 충돌 처리: backend common Source of Truth와 Task Source of Truth가 충돌하면 구현·리뷰를 진행하지 않고 사용자 결정 또는 규칙 정리를 요청한다
- 선행 산출물 경로
- 출력 파일 경로
- 체크포인트 파일 경로
- 역할별 지시

## Role Output

필수 내용:

- 상태
- 변경 또는 판단 요약
- 생성/수정 파일
- 검증 결과
- 사용자 질문 또는 남은 위험

## Checkpoint

필수 내용:

- 현재 단계
- 완료된 작업
- 남은 작업
- 다음 호출이 이어받아야 할 주의사항

정상 완료 시에도 checkpoint를 남긴다. 체크포인트 신호만 보내고 파일을 남기지 않는 것은 실패다.
